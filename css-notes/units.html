Units

Pixels, em, rem, and percentages! Which to use? Check out Josh Comeau's blog about pixels and accessibility.

"When picking between pixels and rems, here's the question you should be asking yourself: 'Should this value scale up as the user increases their browser's default font size?'' This question is the root of the mental model I use. If the value should increase with the default font size, I use rem. Otherwise, I use px." -Josh Comeau

Use rem for font sizes, because using pixels overrides a user's font size choice, and is therefore less accessible for people with poor vision. Using rem means the font size will scale with whatever font size is set as their default. Default is 16px but user's can change it.

Use rem for media queries as well. This may seem counterintuitive, because the mobile layout may end up being viewed on a laptop if the user's default font size is bigger than normal, but that's actually a good thing. If they need the font to be large, the mobile layout will look best.

Using rem with height and width can also be a good idea, because it will preserve buttons and other elements' proportions in relation to the font size within them, however, it can also make elements become too big to fit on mobile, if the user font size is increased too much. To prevent this, set the element's max-width to 100%, so it won't get bigger than the size of the screen.

But using rem on padding and margins means those will also increase if the user increases their default font size, so it might be best to use pixels for padding and margin. The exception would be vertical margins, if you want the space to increase between lines to maintain readability there.

Still not sure which to use? Test it. Change your browser's font size and play with rem and pixels and see which makes the best user experience.

There's also vh (viewing height) and vw (viewing width) that are pretty handy to fill the screen, but a word of caution, vh DOES NOT work well on mobile and is best to avoid.

Images

Images are replaced items that are embedded in the DOM. They aren't actual elements in the DOM. This can make images do unpredictable things when styling them. It's a good idea to wrap images in a div and style the div instead.

Another great way to wrap an image is with the <figure> HTML element. The figure element is a block element that works great with <figcaption>.

The browser thinks of images as text, so it will automatically put a few pixels of line height beneath images. If you don't want it there, set images to display: block or to line-height: 0.

HTML is space-sensitive, so it will put whitespace between side-by-side images. If you don't want it there, remove spaces between the side-by-side <img> tags.

Width

You can set width by using rem (recommended), pixels, and percentages.

With percentages, it's important to remember the width will be calculated based off of the parent's container, so if your div is set to 50% width, it will take up half of the available width of the parent.

    Width Keywords

  • Width: min-content

    Calculates width based off the child element. If text, it will cause it to wrap to be as small of a width as possible. If a block element, it will shrink to fit child.

    Width: max-content

    Also bases width on child element, but max-content never wraps. That means your text will simply run off the page and not be visible.

    Width: fit-content

    Fit-content is magic. It also bases width on the child element, like both of the above, but whereas min-content shrinks text to one word on a line, and max-content cuts off the excess that doesn't fit on one line, fit-content is smart. It will allow the words to fill up the line, then wrap when needed. Yay.

Height

Most of the time, setting the height isn't desirable. We want our pages to be able to expand for any size of content.

But there's one case you might want to set a height, and that's if you only have a tiny amount of content on a page, but you still want that container to fill the screen. You might think to use 100 vh (viewing height), but it doesn't always work exactly. The answer is min-height, but min-height only works if you set a height for the parent containers. That's because the height of the <body> element defaults to shrink to the size of the children. You need to set height: 100% on every element before your container you wish to set a min-height on, including <html> and <body>, then set min-width: 100% in order for that element to fill the viewport height.

It's important to note that Flexbox and Grid do not fix this height issue. So get used to setting height: 100% on <html> and <body> as well as any other parent containers in order to use min-height. Use this in combination with Flexbox to create footers that will stay at the bottom of the page.

Caution: Maybe Just Don't Do These Things

"Exercise caution! It's super important that you never apply the following CSS: button { outline: none; } This rule is sometimes used to hide the blue or black ring that can accompany interacting with a button or link. Sometimes, designers ask us to do this. The problem is that this totally breaks navigation for keyboard users; that ring is required for them to know which element is currently focused! The only exception is if we provide a suitable alternative. For example: button { outline: none; } button:focus { background: navy; color: white; }" -Josh Comeau
